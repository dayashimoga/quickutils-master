# Testing Guide

This document outlines the testing strategies and validation commands for the QuickUtils directory ecosystem. Maintaining 100% pass rates and >90% code coverage is a strict requirement for all deployments.

## The Testing Framework
We use **pytest** as our primary testing framework, augmented with `pytest-cov` to monitor line coverage.

## Running Tests Locally
Tests can be executed concurrently or locally per-project. 

### 1. Running the Global Suite
From the root repository (`quickutils-master`):
```bash
pytest tests/ --cov=scripts --cov-report=term-missing
```
**Expected Output:** This validates core utilities in `scripts/utils.py`, `scripts/build_directory.py`, template parsing, and file sync logic. Coverage must report >90%.

### 2. Running Project-Specific Suites
To ensure that a specific project's `database.json` and generated templates are structurally sound:
```bash
pytest projects/boilerplates-directory/tests/ -vv
```
**Assertion Failures:** If an assertion fails here, it is usually because a generic template was passed missing flags from the project's `conftest.py` (e.g., missing `enable_adsense`).

## Local Link Verification
Whenever new items are added, broken links can ruin SEO. To prevent this, a local verifier crawls the generated `dist/` directory before deployment.
1. Build the project locally first: `python scripts/build_directory.py` (inside the appropriate project directory, or orchestrator root).
2. Run the internal link verifying script: `python scripts/check_links_local.py`
3. Resolve any `404` warnings generated by the verifier before pushing to git.

## Continuous Integration (CI)
GitHub Actions automatically runs the global standard tests globally on every Pull Request and Push event. The pipeline utilizes action constraints to stop Cloudflare deployment triggers if the tests drop below 90% or contain a single failure.
